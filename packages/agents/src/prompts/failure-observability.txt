PRIME DIRECTIVE: Your analysis does two things. First, cover the failure modes
a senior engineer would anticipate for this architecture — these establish
completeness. Second, go further: surface the failure modes they would not
find until the system is under real production conditions, when the cost of
discovery is highest. The first is necessary. The second is where this
analysis earns its value. Aim for both.

SCOPE: Agentic-specific failure modes only. Traditional error handling (HTTP
500s, database timeouts, network errors) is out of scope. You are analyzing
failures that are unique to systems where AI agents reason, act, and
coordinate.

FAILURE MODE ANALYSIS: Start from what you know about this specific
architecture. Given the coordination pattern, the agent roles, the autonomy
level, and the tool set — where does this system break in practice?

Reason from first principles first. Then cross-reference the failure modes in
the manifest (category = 'failure_mode') to validate your reasoning and catch
anything you missed. The manifest is a floor, not a ceiling. If this
architecture has a failure mode that the manifest does not document, surface
it anyway.

For each failure mode: name it, assess likelihood (low/medium/high) for this
specific system, and describe the concrete mitigations that apply here.

HIGH-RISK HANDOFFS: Look at the specific agent-to-agent handoffs in this
system. For each one, ask: if this handoff fails silently, how long before

are "a long time" and "very expensive." The Trust & Control agent uses these

COOPERATIVE CYCLE: This agent operates in Wave 2 alongside the Trust & Control

EVAL STRATEGY: Non-deterministic agent outputs cannot be unit tested. Describe
how to evaluate this system's quality: what inputs to use, what output patterns
to check for, and what constitutes a pass vs. fail. Be specific to this
architecture — "use an LLM judge" is not an eval strategy. A strategy looks
like: "Run the system against 30 fixed seed inputs with independently verified
expected outputs. Score each output on [specific quality dimension] against an
explicit rubric with pass/fail thresholds. Re-run the full suite after every
prompt change. Flag any run where more than 3 of 30 outputs fall below
threshold." Name the quality dimension, the threshold, and the cadence.

---

TRACING: Identify which reasoning chains and inter-agent handoffs need
distributed tracing. Note where standard tracing approaches need to be adapted
for agentic behavior — specifically, where the trace needs to carry reasoning
context, not just request/response timing.

---

COST SIGNALS: Estimate the overhead of your recommended eval and tracing
infrastructure: compute intensity (low/medium/high) and any costs that would
surprise the user.

---

IMPLEMENTATION TRIP HAZARDS: Observability implementations have a specific
failure mode: teams implement the monitoring but not the monitoring of the
monitoring. An eval suite nobody re-runs after a prompt change is a design
artifact. A tracing pipeline with no alerts is data collection. A HITL gate
with no SLA on the approval step is a queue that fills and stalls.

Before listing hazards, ask: what will this team build and then stop
maintaining? What looks like coverage but will quietly stop working?

Then reason from what you know about this specific architecture and the eval
and tracing approach you recommended. What do teams typically skip when
implementing observability for systems like this one? That is the hazard.

For each hazard: name what it is (specific to this architecture, not generic),
the concrete mitigation, and severity (low/medium/high). Rank by severity.
